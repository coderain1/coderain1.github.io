Android性能优化（一）启动优化

应用在冷启动之前要执行三个任务

  • 加载启动APP
  • APP启动之后立即展示一个空白的window
  • 创建APP的进程

然后会执行

  • 创建APP对象
  • 启动main thread
  • 创建启动的activity对象
  • 加载View
  • 布置屏幕
  • 进行第一次绘制

一旦APP完成第一次绘制，系统进程就会用main activity替换已经展示的background window

冷启动（三种启动加速方式）

  • 利用提前展示出来的window，快速展示出来一个界面，给用户快速反馈的体验
  • 避免启动时做密集沉重的初始化
  • 定位问题，避免I/O操作，反序列化，网络操作，布局嵌套

java反射

2018-08-21

java反射

Java反射机制是在运行状态中，对于任何一个类，都能够知道这个类的所有方法和属性；对于任何一个对象，都能调用他的任何一个方法和属性；这种动态获取信息和动态调用对象的方法的功能成为Java的反射机制

  • Class c1 = Code.class
  • Class c2 = code1.getClass()
  • Class c3 = Class.forName(“类的路径”)
  • 获取成员方法method
  • 获取成员变量field
  • 获取构造函数 Constructor

获取成员方法信息

  • getDeclaredMethod（String name,） 获得该类的所有的方法，不包括父类的
  • getMethod 获得该类所有的public方法，包括父类的

    获取成员变量信息

  • getDeclareField(String name) 获得该类自身声明的所有变量，不包括父类的
  • getField(String name) 获取该类所有的public成员变量，包括父类的
  • field.setAccessible(true)//设置允许访问，因为成员变量为private的，所以要手动设置允许访问

RxJava简介

2018-08-21

在Android studio中使用Rxjava,添加gradle依赖

RxJava中的观察者和被观察者

  • 被观察者 （observable）
  • 观察者 （observer）
  • 链接被观察者和观察者（subscribe（））

    ObservableEmitter与disposable

  • ObservableEmitter Emitter发送三种类型的事件（onNext（T value） onComplete（） onError（））
    发送事件

满足的规则

  • observable可以发送多个onNext（）事件
  • observable在发送onComplete（）或者onError（）后还可以继续发送事件，但是observer在接收到onComplete（）或者onError后不再接收事件
  • 上游可以发送多个onComplete但是不能发送多个onError事件

dispose（切断水管）

切断水管后上游可以继续发送事件，而下游接收不到事件
Disposable.dispose()

Rxjava的线程控制

observable和observer默认情况下工作在同一个线程中
通过rxjava的线程控制器
observable.subscribeOn(Schedulers.io/schedulers.newThread()).observeOn(AndroidSchedulers.mainThread())

  • subscribeOn()指定observable发送事件的线程
  • observeOn（）指定observer接收事件的线程
  • 多次指定subscribeOn（）的线程只有第一次指定的线程有效
  • 多次指定observeOn（）的线程，每次指定observer接收事件的线程就会切换一次线程

Rxjava处理嵌套网络请求

Map操作

map可以将observable发送的事件转化为任意的类型给observer接收
flatmap将observable发送的一个事件变换为多个发送事件的observables，然后合并为一个observable发送给下游，但是flatmap不保证发送的顺序
concatMap和flatmap的功能一样，但是concatmap保证顺序

通过flatmap来进行嵌套网络请求的封装

zip操作

通过一个函数将多个observable发送的事件结合到一起，然后发送组合到一起的事件，按照严格的顺序应用这个函数，只发送和发送事件最少的那个observable一样的数据

zip的应用是在：比如一个用户的信息需要两个接口才能获取完，这个时候用zip来组合两个接口请求，就可以同时获取用户的额所有信息

backpressure 背压

当observable发送事件的速度大于observer接收事件的速度，就会出现多余的事件存储到observable的存储队列里面，当多余事件不断增加时就会出现oom的异常
背压就是处理这种异常的方法

被观察者：Flowable
观察者：Subscriber
订阅连接：subscribe（）

ConstraintLayout

2018-07-24

ConstraintLayout

相关网址：https://www.jianshu.com/p/50debc330e91
和Relativelayout一样都是为了减少多层嵌套布局，为约束布局

三种约束

  • 子控件和子控件之间的约束
  • 子控件和父控件之间的约束
  • 子控件和guideline之间的约束

相对定位（Relative positioning）

layout_constraintXXX_toYYYof

XXX:控件本身的边界
YYY:约束控件的边界

当约束控件为GONE时的margins

layout_goneMarginXXX

当约束控件为GONE时，控件自身的marginXXX失效，goneMarginXXX会覆盖marginXXX

居中定位

当控件的约束为 “不可能”的约束

此时button会居中展示

倾向（bias）

  • layout_constraintHorizontal_bias (0最左边 1最右边)
  • layout_constraintVertical_bias (0最上边 1 最底边)
  • bias支持小数定位

尺寸约束（dimensions constrains）

当Constraintlayout为wrap_content时，

  • minWidth
  • minHeight

控件的尺寸约束（widgets dimensions constrains）

  • 确定的尺寸
  • wrap_content
  • 0dp 相当于match_constraint
    match_parent在Constraintlayout中不再被支持

比例（Ratio）

只有一个方向的约束

比例值有两种

  • 宽度和高度之间的比例值（2，0.5）
  • 宽度和高度之间比例（2:1）

宽度和高度都被约束

layout_width和layout_height都为0dp
希望一个维度约束另一给维度添加W或者H进行约束
此时的比例为高度:宽度

链条

创建链条

如果一组小部件通过双向连接，将构成一个链条

链条的样式（chain style）

  • spread chain
  • spread inside chain
  • weighted chain
  • packed chain
  • packed chain with bias

guideline

guideline只能用于constraintlayout中，不会被显示，仅仅用于辅助布局

  • vertical的guideline，宽度为0，高度为constraintlayout的高度
  • horizontal的guideline，高度为0，宽度为constraintlayout的宽度

定位guideline的三种样式

  • 指定左侧或者顶部的固定距离（layout_constraintGuide_begin）
  • 指定右侧或者底部的固定距离（layout_constraintGuide_end）
  • 指定在父控件中的宽度或高度的百分比（layout_constraintGuide_percent）
